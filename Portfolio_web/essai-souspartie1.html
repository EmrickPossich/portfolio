Essai - Evaluer la cause racine d’un dysfonctionnement

Dans le cadre de mon alternance en BUT GEII, j’ai été sollicité pour analyser un dysfonctionnement critique observé sur une toute nouvelle gamme de produits, le PAC6. Ce boîtier, composé d’une carte CPU capable de faire fonctionner des systèmes d’exploitation comme Windows et d’une carte dotée de slots BUS passifs pour l’ajout de modules, venait d’être produit pour la première fois dans le cadre d’une commande de 13 unités. Le problème a été détecté au moment du test final, alors que le produit semblait initialement fonctionnel

Le bug rencontré était le suivant : lors d’un redémarrage de Windows, le système pouvait se bloquer totalement, rendant nécessaire un redémarrage manuel du boîtier. Ce comportement erratique posait un risque majeur en termes de fiabilité. David Ouattara m’a confié la mission de mener des tests pour identifier la cause racine du dysfonctionnement.

Des premiers éléments d’enquête m’ont été fournis : le bug apparaissait de manière aléatoire lorsque le clavier était connecté, uniquement sur le site de test final à rue Émile Zola, et non sur le site de Bruyères où se situe mon laboratoire. J’ai donc choisi de me rendre directement à Zola afin d'observer le phénomène dans ses conditions réelles d’apparition. J’y ai conçu un plan de test structuré, en variant les périphériques (deux claviers, deux souris) et les conditions de test (paillasses différentes, sources d’alimentation variées). Chaque configuration était soumise à 50 redémarrages si aucun bug ne survenait.

Les résultats ont rapidement mis en évidence que ni la paillasse, ni la référence du clavier, ni le réseau électrique ne semblaient avoir d’impact significatif. Le bug survenait de façon similaire, quel que soit le clavier, souvent en moins de 10 redémarrages. J’ai même observé des cas où le simple branchement d’un clavier déclenchait le problème. Cela m’a conduit à suspecter une influence des périphériques USB sur les charges ou la communication.

Pour écarter l’hypothèse d’un lien avec le site, j’ai refait exactement les mêmes tests à Bruyères, en reprenant tous les éléments (claviers, souris, alimentation, écran) de Zola. Le bug est réapparu, confirmant que le lieu n'était pas en cause. J’ai ensuite procédé à des tests avec des charges USB simulées, pour maximiser la consommation sur les ports, mais aucun dysfonctionnement n’est apparu, ce qui a permis d’écarter l’hypothèse d’un problème lié à l’intensité.

J’ai alors entrepris une campagne de tests combinatoires poussée, pour identifier les éléments déclencheurs précis. La figure 1 illustre ces tests : j’ai testé les différentes combinaisons de périphériques (clavier seul, souris seule, les deux, aucun), de blocs d’alimentation, de câbles HDMI et d’écrans. Chaque scénario était soumis à 50 redémarrages. Les résultats ont révélé que le bug dépendait uniquement de la référence d’écran et de la présence de périphériques USB. Lorsque l’écran de référence 2 était utilisé ou qu’aucun périphérique n’était connecté, le bug n’apparaissait jamais.

Cette découverte a permis de cibler la carte CPU comme unique composant susceptible d’être responsable du dysfonctionnement, celle-ci étant la seule variable constante, fabriquée par un fournisseur externe. Malheureusement, je n’avais pas accès à d’autres exemplaires de cette carte pour valider s’il s’agissait d’un défaut de fabrication ou de conception.

Pour finaliser le diagnostic, j’ai établi une dernière série de tests basée sur un large échantillon d’écrans connectés en HDMI, dans la configuration la plus sujette au bug (clavier 1 + souris 1). La figure 2 présente les résultats : certains écrans provoquent le bug systématiquement, d’autres jamais. J’ai également testé un adaptateur HDMI-VGA (aucun bug) et un amplificateur HDMI (bug réapparu), ce qui renforce l’idée que la communication via HDMI joue un rôle critique dans l'apparition du problème.

Grâce à l’ensemble de ces tests, j’ai pu localiser précisément les conditions d’apparition du bug, en excluant systématiquement les causes non pertinentes. Même si une correction technique n’était pas réalisable dans l’immédiat, mon travail a permis à David Ouattara de prévenir le client du risque potentiel selon le type d’écran utilisé. Il servira également de base de test pour les 12 autres boîtiers PAC6, afin de vérifier si le dysfonctionnement est récurrent ou isolé.
